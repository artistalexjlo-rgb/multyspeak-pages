> Как читать: каждый этап = **Цель** · **Действия** · **🚦 Гейт** (условие, без которого
> дальше нельзя) · **📊 Пусто/густо** (по чему ты сам оцениваешь, тонко или плотно).
> Правила качества — в `BUILDER_RULES.md`, тон — в `MESSAGING.md`. Двигаемся по гейтам,
> не по ощущению «вроде готово». Гейт не пройден → стоп, чиним, не идём дальше.

Бренды: портал = **MultySpeak** (суть, домен), ассистент = **Luky** (имя на CTA).

## ЭТАП 3. Автоматизировать узел-экстрактор (сердце билдера)

**Цель:** превратить мой РУЧНОЙ экстракт в воспроизводимый прогон по всей ячейке.
**Действия:**
- промпт экстрактора (Claude): на вход — все `ai_lesson` ячейки, на выход — page-data JSON
  (short_answer + FAQ), строго grounded, с подсчётом провенанса по ВСЕМ паттернам (не выборке);
- встроить фильтр шума (выкинуть мис-теги, как 4 «Shopping» в Finance);
- кластеризация ячейки на под-вопросы (bge-m3 на тексте `ai_lesson`, не на теге).
**🚦 Гейт перехода (главный гейт качества):**
- авто-страница по `br/Finance` **не хуже ручного эталона** (слепое сравнение: плотность,
  grounded, читаемость) — это объективная планка, эталон уже есть;
- **0 выдуманных фактов** при проверке против источников (спот-чек N фактов → есть в данных);
- провенанс на каждый факт ≥2, тонкие выкинуты;
- текст связный (не телеграф).
**📊 Пусто/густо:** если авто ≥ эталон на чистой ячейке — узел готов, можно масштабировать.
Если авто жиже/врёт — стоп, узел не готов, дальше НЕ идём (иначе тиражируем брак). Это
самый строгий гейт: проходит он — проходит весь билдер.

**🔧 МЕХАНИЗМ (выбран): ролевой конвейер субагентов писатель → чекер → редактор.**
- **Писатель** — на вход все `ai_lesson` ячейки, на выход grounded page-data
  (short_answer + FAQ), живой прозой (не телеграф).
- **Граунд-чекер (adversarial)** — по КАЖДОМУ факту: «есть в источниках? в каком паттерне?».
  Нет пруфа → факт вон. Отдельная роль именно против галлюцинаций (писатель «дописывает гладко»).
  Тут же ловит мис-тег («это вообще про эту тему?» → 4 «Shopping» в Finance).
- **Редактор** — против `BUILDER_RULES`/`MESSAGING`: провенанс ≥2, тонкое выкинуто, тон,
  читаемость, плотность. Возвращает правки или REJECT + короткий вердикт.
- **⛔ КАП РАУНДОВ: максимум 2 прохода редактора.** Не вылизывать до бесконечности — после
  2-го прохода либо pass, либо ячейка помечается «не дотянули» и откладывается (не блокирует батч).
- Отчёт от агентов — НЕ простыни: редактор отдаёт краткий вердикт (pass/fail + провенанс-статы
  + 1-2 строки проблем). Конвейер воспроизводим по ячейкам → это и есть масштабируемый узел.

**💰 ПРОД-КОНФИГ УЗЛА (подтверждён тестом + пробой, против «разоримся»):**
Тест 9 ячеек на сильной модели = ~983K токенов (~109K/стр) — это ВЕРХНЯЯ граница, не прод-цена
(полное трио, 3× перечитка исходника). Безопасность НЕ должна зависеть от ума писателя
(даже сильная модель галлюцинировала: vn/finance выдумал курс Сбера «335-342» → чекер вырезал).
Поэтому ум переносим с генерации на проверку:
- **Писатель = gemini-3.1-flash-lite** (RPD 500/ключ × 12 ключей, генерационная квота ПРОСТАИВАЕТ —
  ~60-80/день; эмбед-квота отдельная и занята вики, писатель её НЕ трогает). Бесплатно.
  Проба 2026-06-24: на УЗКОЙ задаче (сожми заданные сниппеты + процитируй заданные ID) flash-lite
  держит JSON-формат, цитирует валидные ID, ноль выдумок. Старый «narrate-not-execute» не выстрелил —
  потому что работа узкая. Нюанс: играет осторожно-обобщённо (уходит от чисел) → промптом «вставляй
  конкретные числа из сниппетов».
- **Предкластеризация = bge-m3 локально (free).** Он группирует ячейку в под-вопросы И отбирает
  2-5 паттернов-кандидатов с ID под каждый. Писателю остаётся тривиальное «перепиши данное» —
  слабость flash-lite заперта, цитата почти детерминирована (ID заданы заранее).
- **Верификатор чисел/имён = детерминированный код (free).** Процитированный ID реально содержит
  это число/название? Нет → факт режется. Ловит «Сбер-335» класс без LLM и без оглядки на тир писателя.
- **Чекер (ум, сильная модель) = только по СМЫСЛОВОМУ остатку** (перефразированная ложь, что
  детерминированная проверка не закрыла) + сэмпл. Не по всей странице. Вот сюда и тратимся — малая доля.
- **Пейсинг:** free-tier flash-lite жёсткий по RPM — батчи + sleep, считать RPM до запуска
  ([[feedback_pace_api_calls]]). Корпус курированный (сотни стр, не десятки тысяч) → RPD хватает за дни.
- **Дельта-свежесть:** пересобирать только кластеры с новым (манифест), не весь корпус.

## ЭТАП 4. Вскрыть лонг-тейл (ров) — суб-кластеризация помоек

**Цель:** найти узкие смачные пары внутри широких бакетов (`Travel` 40%), где конкуренция
низкая И только в наших чатах есть ответ.
**Действия:**
- bge-m3 кластеризация `Travel` (и др. помоек) по смыслу → под-темы;
- по каждой под-теме оценить: данные есть? (provenance) × топы воюют? (груб. оценка SERP/вручную);
- отобрать пары «узкий интент × grounded × низкая конкуренция» — приоритет публикации.
**🚦 Гейт перехода:**
- из помоек выделены ≥N связных под-тем с провенансом ≥2;
- список лонг-тейл-кандидатов, размеченных по 2 осям (где именно ров).
**📊 Пусто/густо:** если суб-кластеры дают узкие темы с данными — это золото, печатаем.
Если помойка не кластеризуется в осмысленное (мусор/слишком общее) — значит там и нет рва,
не тратим. Сигнал прямой: связные плотные под-кластеры = густо, размазня = пусто.

## ЭТАП 5. Пакетная генерация (масштаб)

**Цель:** прогнать узел по очереди ячеек из Этапа 2 + лонг-тейл из Этапа 4.
**Действия:** батч-прогон экстрактора по top geo×topic + отобранным лонг-тейл; рендер; сборка сайта.
**🚦 Гейт перехода (на КАЖДУЮ страницу, не на батч):**
- провенанс ≥2, 0 выдуманных, связный текст;
- широкая тема: ≥5-6 плотных FAQ + short_answer; лонг-тейл: ≥1 тугой grounded-ответ;
- тонкое-без-данных НЕ публикуется (паркуется до накопления);
- 0 битых ссылок, все чипы/линки резолвятся (или честно «скоро»).
**📊 Пусто/густо:** доля страниц, прошедших гейт, из попыток. Высокая доля = ячейки густые.
Много паркуется = добрались до тонкого края корпуса (нормально, не брак — ждём граб/sweeper).
**⚠️ Не молчать о пропусках:** логировать, сколько ячеек спарковано и почему (тонко/шум) —
иначе «сгенерили 200 страниц» читается как покрытие, которого нет.

## ЭТАП 6. Деплой-конвейер + свежесть

**Цель:** не разовый прогон, а живой конвейер (новое награбленное → обновлённые страницы).
**Действия:**
- пересборка по расписанию (та же феррари-логика: движок сам, оператор в петле, обратимо);
- sweeper уже кормит 2-й контур → новые паттроны → страницы пересобираются;
- уведомление о прогоне/изменениях → тех-канал дайджеста (чирик, уже чинён).
**🚦 Гейт перехода:**
- прогон воспроизводим одной командой/по расписанию, не «руками каждый раз»;
- видно ЧТО изменилось (diff страниц), чирик в канал работает;
- откат возможен (страницы версионируются в git репо pages).
**📊 Пусто/густо:** если новое в grab за неделю реально доезжает до страниц — конвейер живой.
Если страницы застыли, а данные растут — конвейер мёртвый (фантом), чинить.

## ЭТАП 7. Замер и воронка (доказать смысл)

**Цель:** убедиться, что трафик идёт И перетекает в продукт.
**Действия:** Search Console (индексация, показы, позиции по лонг-тейл); счётчик кликов CTA→Luky.
**🚦 Гейт:**
- страницы индексируются, есть показы по целевым лонг-тейл-запросам;
- видно конверсию страница→открытие Luky (хоть грубо).
**📊 Пусто/густо:** показы по лонг-тейл растут = ров реальный. Ноль показов при индексации =
темы никому не нужны (пере-оценить выбор ячеек). Клики в продукт = воронка работает; ноль =
CTA/оффер не цепляет, чинить мессадж (не билдер).

## ИНВАРИАНТ НАВИГАЦИИ — «СКОРО» не должно протухать

**Проблема:** плитка/чип «СКОРО» = обещание, что страница едет. Если тема висит «СКОРО»
вечно (никогда не собирается) — это ложное обещание, оно протухает, и никто не сверяет.

**Правило (by design, чтоб протухание было НЕВОЗМОЖНО):**
1. **«СКОРО» = только data-backed.** `wire.py` ставит soon-плитку ТОЛЬКО для `geo×topic`,
   у которых реально есть данные выше гейта спроса (тема действительно в очереди). Нет данных →
   плитки нет вовсе. НЕ обещать из статического каталога вслепую. → значит «СКОРО» честно значит
   «уже едет».
2. **Состояние — в ДАННЫХ, не в памяти.** Что обещано/в очереди/собрано деривируется из манифеста
   buildable-ячеек (VPS `extracted_patterns`) + рендер-манифеста, самосверяется на каждом `wire.py`.
   НЕ держать «что обещали» в памяти Claude — она протухает (см. [[feedback_trust_context_not_memory]]).
3. **Сторож сверяет протухание** (обязанность cron-аудитора, [[process_autonomous_auditor_bounds]]):
   - «СКОРО» с данными, но без страницы N дней → билдер застрял → чирик «собери» в тех-канал;
   - «СКОРО» без данных (демоут) → убрать обещание.

**Статус:** сейчас `wire.py` берёт СТАТИЧЕСКИЙ каталог из 7 тем → обещает вслепую (дыра, поймал юзер).
- [ ] `wire.py`: soon-плитки только для data-backed тем (сверка с наличием данных на VPS).
- [ ] cron-аудитор: пункт «протухшие СКОРО» в ежедневную сверку.

- `status.json` — {state, built, skipped, failed, gate_drops, last_cell, pending, ts} (glance).
- `audit.log` — построчно что собрано/сколько FAQ/что focus+gate выкинули.
- `STOP` (touch-файл) — грациозный стоп batch (не pkill).
- чирик в тех-канал (`chirp.sh`) ТОЛЬКО при сбое/STOP; чистый успех — тихо.

**Cron-сторож (`auditor.sh`, crontab `30 8 * * *` = 11:30 МСК):**
- собирает status+audit+gemini_quota+живость batch → `claude -p` ТОЛЬКО судит (текст→текст,
  БЕЗ инструментов = физически не зациклится/не запустит) → чирик в дайджест только при `ALERT:`.
- flock от наложений; `audit_sentinel.log` — лог решений. Тест 2026-07-03 → «OK» (здоров, молчит).
- Границы соблюдены: read-only, 1×/сутки (без само-поллинга), только эскалация (газа нет),
  один cron (без само-размножения). См. [[process_autonomous_auditor_bounds]].
